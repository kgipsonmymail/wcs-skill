## 核心锚点：WCS 的目标与定位

### 定位

> **统领，不执行**

- WCS **不是**执行开发的 skill
- WCS **是**开发任务的**统领**：启动任何开发任务时第一个使用
- 它帮助 AI 找到合适的 skill/doc/file，然后让相应的 skill 去执行

### 核心目标

> **通过架构设计协助 AI，最节省 token 和上下文长度（精准），发挥 AI 最大能力，最快解决开发任务**

### 三层上下文维度

```
skill（技能层）  →  doc（文档层）  →  file（文件层）
     ↑                ↑                ↑
  工具能力          项目知识          源代码
  按需激活          按需阅读          按需定位
```

**WCS 负责精准调度这三层上下文。**

### WCS 强制的能力

| 能力 | 目的 | 保证 |
|------|------|------|
| **代码规范执行** | 良好代码质量 | 强制 AI 执行 |
| **项目迭代保证** | 与项目保持内容一致性 | 强制 docs 更新 |
| **交接保证** | 上一轮和下一轮开发能安全交接 | git备份 + 文档 + 目录结构 |

## 一、问题陈述

### 当前 WCS 阅读顺序的问题

**现状**：新开对话 → AI 阅读所有 docs → 开始执行任务

**问题**：
- 每个任务都要读全部 docs，效率低
- docs 数量增长后，上下文爆炸
- AI 无法快速定位与当前任务相关的文档
- 不同类型任务（功能开发/bug修复/架构调整）需要不同的文档子集

### 期望的三层管理架构

```
┌─────────────────────────────────────────────────────────────┐
│  中枢文档（YAML/项目总览）                                     │
│  - 项目全局索引                                               │
│  - 文档位置映射                                               │
│  - AI 决策依据（看哪些 docs）                                  │
└──────────────────────┬──────────────────────────────────────┘
                       │ 引用
┌──────────────────────▼──────────────────────────────────────┐
│  docs 目录（按主题/模块组织）                                   │
│  - dev_plan.md / dev_log.md / features.md                    │
│  - 模块级文档（backend.md / frontend.md / database.md）         │
│  - AI 按需选择性阅读                                           │
└──────────────────────┬──────────────────────────────────────┘
                       │ 引用
┌──────────────────────▼──────────────────────────────────────┐
│  全文件（源代码）                                              │
│  - AI 可直接定位到具体文件/函数                                 │
└─────────────────────────────────────────────────────────────┘
```

**核心优化**：AI 收到任务后，**先读中枢文档** → 决定需要哪些 docs → 按需阅读

## 二、参考系统研究

### 2.1 CodeGraph

**功能**：代码知识图谱，通过 AST 分析建立代码结构索引

```
代码文件 → AST 解析 → 符号索引 → 关系图
                            ↓
                      MCP Server 提供查询接口
```

**可借鉴**：
- 用程序维持对所有文件的感知（增量索引）
- 关系映射能力（函数调用链、依赖关系）
- 查询接口（按符号/关系检索）

**不采用**：
- 不做 AST 级别的代码分析（重、依赖 Node.js）
- 不 fork 或二次开发（成为直接使用者）

### 2.2 Context-Mode（RTK 相关）

**功能**：上下文压缩，据说节省 98% tokens

**可借鉴**：
- 按任务裁剪上下文的思路
- 主动决定"看什么、不看什么"

**不采用**：
- 不安装基础依赖（Node.js 环境、RTK Rust 工具链）

### 2.3 RTK（Token Guard）

**功能**：Eagle Mem 的 token 保护，监控 shell 输出 token

**结论**：重依赖（Rust 工具链），暂不考虑

## 三、依赖优先级框架

按你的优先级排序：

| 优先级 | 方案 | 说明 |
|--------|------|------|
| 0 | **纯自管理**（0依赖） | 用 YAML/纯文本做中枢，不装任何外部工具 |
| 1 | **轻依赖** | 加一点功能就能实现最多作用（如用 sqlite 做索引） |
| 2 | **现成项目** | 成为 codegraph 的直接使用者，不 fork/二次开发 |
| 3 | **基础依赖** | codegraph 所依赖的相关配置（Node.js 等） |

## 四、YAML 中枢设计（已实现）

### 4.1 中枢文件位置

```
docs/project_index.yaml
```

### 4.2 中枢结构

```yaml
# 三层结构
project:                    # 项目元数据
skills:                     # Skill 层：按需激活
docs:                       # Doc 层：按需阅读
files:                      # File 层：按需定位
task_contexts:              # 任务上下文推荐
```

### 4.3 核心字段

| 字段 | 用途 |
|------|------|
| `project` | 项目基本信息 |
| `skills` | 本地 skill 索引（path, purpose, tags, active_when） |
| `docs` | 项目文档索引（path, purpose, read_when, tags） |
| `task_contexts` | 任务类型 → 推荐 skill + doc 映射 |

### 4.4 AI 执行流程

```
任务输入
  ↓
读取 project_index.yaml
  ↓
根据 task_contexts 确定推荐
  ↓
激活 skill → 阅读 doc → 定位 file
```

## 六、WCS 发展脉络与自我迭代问题

### WCS 发展四阶段

| 阶段 | 形态 | 核心 | 问题 |
|------|------|------|------|
| **阶段一** | 提示词 | 代码规范流程 | 散落在对话中，无法复用 |
| **阶段二** | Skill | 一次安装全项目生效 | docs 维护依赖自觉，难以持久 |
| **阶段三** | 强化 docs | 严格 docs 体系 | AI 每次都读全部 docs，效率低 |
| **阶段四** | 三层架构（当前） | YAML 中枢 + 按需索引 | 尚未实现 |

### 核心待解决问题

#### 问题一：WCS 没有自我迭代能力

**现象**：WCS 不能随着接管的 project 增多而自我提升

**原因**：WCS 是一次性安装的静态规则，不会从使用经验中学习

**期望**：WCS 能在多个项目中运行后，总结出通用模式，反哺到 skill 本身

**可能的解决方向**：
- `docs/discussions/` 积累的项目经验 → 是否可以提炼成新的 skill 规则？
- 每个项目的 `error_book.md` → 是否可以形成 WCS 的"常见错误检查清单"？
- 每个项目的 `features.md` → 是否可以形成 WCS 的"能力矩阵"？

#### 问题二：规则背后的目标更重要

**现象**：规则可能落后，但目标不会

**例子**：
- 规则："每次 commit 前必须跑测试" → 可能过时（AI 时代测试策略变了）
- 目标："保证代码质量，防止回归" → 不会过时

**WCS 应该记录的"目标"**：
1. **质量目标**：不写烂代码，防止回归，便于交接
2. **效率目标**：减少重复劳动，快速定位问题
3. **协作目标**：多 AI/多人都能无缝接续
4. **知识沉淀目标**：错误不白踩，经验可传承

这些目标应该是 WCS 的核心，而规则是达成目标的手段。规则可以演进，目标是锚点。

## 七、WCS 定位与核心能力重构（2026-06-05 深化）

### 定位：统领，不执行

**核心认知转变**：
- WCS **不是**执行开发的 skill
- WCS **是**开发任务的**统领**：启动任何开发任务时第一个使用
- 它帮助 AI 找到合适的 skill/doc/file，然后让相应的 skill 去执行

### 三层上下文维度

```
skill（技能层）  → doc（文档层）  → file（文件层）
     ↑                ↑                ↑
 工具能力          项目知识          源代码
 按需激活        按需阅读          按需定位
```

**核心洞察**：skill/doc/file 是三个不同维度的上下文，WCS 负责精准调度。

### WCS 强制的能力

| 能力 | 目的 | 保证 |
|------|------|------|
| **代码规范执行** | 良好代码质量 | 强制 AI 执行 |
| **项目迭代保证** | 与项目保持内容一致性 | 强制 docs 更新 |
| **交接保证** | 上一轮和下一轮开发能安全交接 | git备份 + 文档 + 目录结构 |

### Errorbook 释放机制（新）

**问题**：errorbook 记录越来越多后，AI 需要全文阅读才能找到相关内容 → 上下文爆炸

**新理解**：
- errorbook 类似 memory，内容是**无关的、散乱的**
- 应该**定时清空**，把内容归纳到系统性文件

**释放流程**：
```
errorbook（散乱记录）
    ↓ 定期归纳
系统性文件：
  - features.md（新能力记录）
  - CODING_STANDARDS.md（规范补充）
  - dev_plan.md（待办补充）
  - project_status.md（状态补充）
    ↓ 释放后
errorbook 清空（类似 memory 重置）
```

### Skill 管理机制（新）

**需求**：WCS 安装后要**管理本地所有 skill**

**设计**：
- WCS 启动时扫描 `~/.hermes/skills/`
- 建立 skill 索引（用途、能力描述）
- AI 按需激活相关 skill，**避免打开不相关 skill**
- 减少上下文窗口

**层级展开**：
```
开发任务输入
    ↓
WCS 分析任务类型
    ↓
精准激活相关 skill（避免全开）
    ↓
skill 展开其需要的 doc
    ↓
doc 引用具体 file
```

### 核心目标（最高优先级）

> **通过架构设计协助 AI，最节省 token 和上下文长度（精准），发挥 AI 最大能力，最快解决开发任务**

所有功能设计都围绕这个目标。
