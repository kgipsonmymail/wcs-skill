> WCS 是 AI 开发方法论框架。每个开发任务第一个激活它，通过**状态识别**决定读什么文档，避免上下文过载，在保证开发质量的前提下最快完成任务。WCS 通过在实践中不断总结经验来迭代自身。

> 通过架构设计协助 AI，最节省 token 和上下文长度（精准），发挥 AI 最大能力，最快解决开发任务。

### 3.2 最小化开场 ✓ 部分达成

| 检查项 | 结果 |
|--------|------|
| project_index.yaml 开场必读 | ❌ **违反**：未在第一时间读取 |
| dev_log.md 延迟加载 | ✓ 按需读取（维护阶段确实需要） |
| 未一股脑加载所有 doc | ✓ 仅读取了 3 个相关 doc |

**问题**：
- WCS 规则明确说 `project_index.yaml` 是"唯一开场必读"
- 但 AI 实际直接读了 `dev_log.md`，没有先读 `project_index.yaml`
- **这违反了 WCS 规则，是错误的**——无论内容是否已知，开场读它是框架的锚点动作

**结论**：没读是不对的。不能因为"内容已知"就跳过。project_index.yaml 的价值不在于内容是否变化，而在于它是整个 WCS 状态识别框架的触发入口。跳过它意味着跳过了"确认当前状态"的强制动作。

**应该优化的是 project_index.yaml 本身**：让它在维护阶段也包含高价值信息（如当前维护重点、最近变更摘要），而不是"因为维护阶段没新内容所以可以跳过"。

**影响评估**：
- 本次影响小，但长期来看，规则被绕过会让 AI 自行判断"什么时候值得读"，破坏 WCS 的结构化设计
- 这不是"规则设计有问题所以绕过"，而是"AI 执行不到位"

## 四、核心发现：WCS 规则的结构性矛盾

### 矛盾点：project_index.yaml 在维护阶段价值不足

**表面现象**：
- WCS 规则要求"唯一开场必读"
- 但 AI 实际直接跳过了它读 dev_log.md
- AI 认为"内容无变化，读了无新信息"

**真正的矛盾**：
- WCS 把它设计成"所有阶段开场必读"
- 但它对维护阶段没有足够高价值的内容
- 当规则和直觉冲突时，AI 选择了跳过

**结论（修正后）**：
1. **没读是违反规则**，不是"合理跳过"
2. **但 project_index.yaml 在维护阶段确实缺乏高价值内容**，这是规则被绕过的根本原因
3. 解决方案不是"允许维护阶段跳过"，而是"让 project_index.yaml 在维护阶段也有高价值"

---

## 五、优化建议

### 建议 1：强化 project_index.yaml 在维护阶段的价值（优先级：高）

**现状**：project_index.yaml 每次都全量读取，但维护阶段 AI 认为"内容无变化，读了无新信息"，导致规则被绕过。

**问题本质**：不是"要不要跳过"的问题，而是"project_index.yaml 在维护阶段是否真的有价值"。

**优化方案**：在 project_index.yaml 中为维护阶段增加高价值字段，例如：

```yaml
# 在 project_index.yaml 增加
maintenance:
  current_focus: "mediary-dev SM18 开发完成，待验证并丰富功能"  # 当前维护重点
  recent_changes:                              # 最近变更摘要（每次维护后更新）
    - date: "2026-06-09"
      change: "mediary-dev SM18 页面 5 Tab 全部完成"
    - date: "2026-06-05"
      change: "Module 1-7 全部落地"
```

**好处**：
- AI 开场必读时，每次都能快速看到"当前维护重点"
- 不需要去 dev_log.md 里翻找
- 维护阶段 project_index.yaml 变成真正的锚点文档，而不是"已知内容"

**注意**：这个优化是为了让规则被遵守——project_index.yaml 应该在所有阶段都有价值，而不是"因为维护阶段没新内容所以可以跳过"。

---

### 建议 2：增加 WCS 规则违反的自我上报机制（优先级：高）

**现状**：Session 反思块有"违反记录"字段，但 AI 经常不填或一笔带过。

**问题**：
- 违反规则时 AI 倾向于"解释原因"而不是"承认违反"
- "合理理由"会让违反变成常态

**优化方案**：
- 将"违反记录"改为强制必填项
- 违反时必须说明：①违反了什么规则 ②原因 ③下次如何避免
- 不允许用"已知内容"等理由绕过规则

---

### 建议 3：区分"硬性规则"和"软性指导"（优先级：中）

**现状**：WCS 所有规则都是强制（"强制执行"、"必须"），但实践中 AI 需要判断力。

**优化方案**：将 WCS 规则分为两类：

| 类型 | 标记 | 说明 |
|------|------|------|
| 硬性规则 | ⚠️ 不可绕过 | 如"commit 前验证"、"禁止二进制入 git" |
| 软性指导 | 💡 建议遵循 | 如"开场优先读 project_index.yaml" |

**好处**：AI 在有合理原因时可以跳过软性指导，但必须遵守硬性规则。减少"规则被系统性绕过"的风险。

---

### 建议 4：为维护阶段设计专门的快速路径（优先级：低）

**现状**：维护阶段每次都要读 project_index.yaml + dev_log.md + project_status.md，但 project_index.yaml 几乎不变。

**优化方案**：
- 维护阶段开场只读 dev_log.md 的"最近 3 条"（通过 offset/limit）
- dev_log.md 维护一个 `[CURRENT]` 标记，AI 只读标记位置之后的内容
- project_index.yaml 在维护阶段变为"变化时读"而非"每次必读"

**注意**：这个优化有风险（可能错过 project_index.yaml 的重要变更），建议搭配建议 1 的 last_modified 机制。

---

### 建议 5：维护前强制 git backup（优先级：高）

**现状**：WCS 自迭代规则说"先 git 备份，再开发新功能"，但这次维护直接跳过了。

**问题**：虽然本次只有文档变更风险低，但如果维护包含代码变更，跳过 backup 可能导致不可回滚。

**优化方案**：在 SKILL.md 的"维护阶段"文档加载顺序中，明确加入：

```
project_index.yaml（开场唯一必读）
  → git status（确认无未提交变更，若有则先 commit）
  → dev_log.md（当前状态章节）
  → project_status.md
```
