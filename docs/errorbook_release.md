# Errorbook 释放机制

## 核心理念

Errorbook 像人类的记忆——内容会过时，积累多了反而是负担。需要定期清空，把有价值的内容归纳到系统性文件。

---

## 触发条件

满足任一条件即触发释放：

| 条件 | 说明 |
|------|------|
| 条目数量 > 30 | 内容太多必然过时 |
| 时间 > 3 个月未释放 | 定期清理 |
| 人工触发 | 用户说"清理 errorbook" |
| 项目重大变更前 | 重构/迁移前整理 |

---

## 释放流程

### 第一步：阅读全部条目

读取 `docs/error_book.md` 所有内容，理解每个条目的：
- 问题本质
- 发生场景
- 解决方案
- 根因

### 第二步：分类归纳

每个条目归入以下类别之一：

| 类别 | 归纳目标 | 操作 |
|------|---------|------|
| 代码规范问题 | `CODING_STANDARDS.md` | 新增规范条目或加强现有规范 |
| 已知 bug | `error_book.md`（保留，标记为 resolved） | 标记为 resolved，附日期 |
| 流程问题 | `workflow.md` 或 `dev_plan.md` | 提出改进建议 |
| 工具/环境问题 | `project_status.md` 或 notes | 记录环境注意事项 |
| 项目特定知识 | `features.md` 或 `structure.md` | 更新对应章节 |
| 无长期价值 | 丢弃 | 直接删除，不心疼 |

### 第三步：更新 error_book.md

- 保留仍有效的未解决问题
- 已归纳到其他文件的，标记 `→ 已归纳到 XXX`
- Resolved 问题保留 30 天后删除
- 释放后条目数应 < 15

### 第四步：记录释放

在 `dev_log.md` 记录：

```
## YYYY-MM-DD - Errorbook 释放

- 释放前条目数：N
- 归纳到 CODING_STANDARDS：X 条
- 归纳到 features：Y 条
- 丢弃（无长期价值）：Z 条
- 保留（仍有效）：W 条
```

## Errorbook 写作规范（减少释放负担）

写 errorbook 条目时遵循：

1. **一句话描述问题**
2. **根因是什么**（不要只写现象）
3. **解决方案是什么**
4. **如何避免再次发生**（一句话）

模板：

```markdown
## [日期] 问题简述

**问题**：...
**根因**：...
**解决方案**：...
**预防**：...
```

## 常见问题

**Q: 有些问题反复出现怎么办？**
A: 反复出现 = 流程/规范问题，不是 personal 经验。应该归纳到 `CODING_STANDARDS.md` 或 `workflow.md`。

**Q: 什么时候 errorbook 保留不释放？**
A: 当问题是项目本身的已知限制（不是代码问题/流程问题），且没有解决方案时。这种情况应改写为 `project_status.md` 中的"已知限制"。

**Q: 释放后会不会丢失有用的上下文？**
A: 会丢失临时的调试过程，但不会丢失有用的经验。经验已经被归纳到了对应文件。原始调试过程本来就不应该长期保留。
