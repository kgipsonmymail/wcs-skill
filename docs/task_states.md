# 任务状态触发机制

## 核心思路

不同任务状态触发不同的规范子集。不再一次性完整阅读所有规范，只读当前状态相关的部分。

触发流程：

```
用户任务 → 匹配 task_context → 激活对应 state_rules → AI 按需执行子集
```

---

## 任务状态 → 触发规则映射

### state: default（新任务/通用）

**触发条件**：无法归类到其他状态时

**激活的规范子集**：
- 阶段 0 完整执行（初始化与对齐）
- 阶段 1 完整执行（开发前置门禁）
- 阶段 2-5 按需执行

**重点关注**：
- 需求是否清晰
- 验收标准是否明确
- 约束条件是否已知

---

### state: bug_fix（Bug 修复）

**触发条件**：`task_contexts.bug_fix`

**激活的规范子集**：
- 阶段 0（定位错误日志、复现问题）
- 阶段 1（制定修复计划 + 回滚策略）
- 阶段 2（聚焦变更，不扩大范围）
- 阶段 3（重点：复现验证 + 回归验证）
- 阶段 4（重点：更新 error_book.md）
- 阶段 5（若涉及多人交接）

**跳过**：features.md 更新（除非真的是新能力）

**重点关注**：
- 根因是否找到
- 是否有回滚方案
- 是否引入新 bug

---

### state: new_feature（新功能开发）

**触发条件**：`task_contexts.new_feature`

**激活的规范子集**：
- 阶段 0-5 全部执行（最完整流程）
- 额外：features.md 更新（必须）
- 额外：dev_plan.md 更新（必须）

**重点关注**：
- 是否符合现有架构
- 是否有破坏性变更
- 是否更新了能力清单

---

### state: refactor（重构）

**触发条件**：`task_contexts.refactor`

**激活的规范子集**：
- 阶段 0（理解当前结构）
- 阶段 1（重点：变更范围界定 + 回滚方案）
- 阶段 2（聚焦：只做重构，不加功能）
- 阶段 3（重点：行为对比验证，确保重构前后行为一致）
- 阶段 4（structure.md 更新）
- 阶段 5（若有必要）

**跳过**：features.md（除非真的变了能力）

**重点关注**：
- 变更范围是否可控
- 重构前后行为是否一致
- 是否有性能回退

---

### state: code_review（代码评审）

**触发条件**：`task_contexts.code_review`

**激活的规范子集**：
- 阶段 0（阅读评审范围内的代码）
- 阶段 1（制定评审计划）
- 阶段 3（评审验证：lint/类型/测试）
- 不走阶段 2（实现）和阶段 4（文档同步，除非发现新问题）

**重点关注**：
- 是否有规范违反
- 是否有安全隐患
- 是否有性能问题

---

### state: handover（交接/接续）

**触发条件**：`task_contexts.handover`

**激活的规范子集**：
- 阶段 0（阅读交接文档）
- 阶段 1（理解上下文）
- 阶段 3（验证当前状态）
- 阶段 5（产出交接块）

**跳过**：阶段 2（除非需要继续实现）

**重点关注**：
- 上下文是否接上
- 假设是否记录
- 下一步是否清晰

---

### state: docs_update（纯文档更新）

**触发条件**：`task_contexts.docs_update`

**激活的规范子集**：
- 阶段 0（确认要更新的文档范围）
- 阶段 4（更新对应 doc）
- 阶段 5（若需要交接）

**跳过**：阶段 1（开发门禁）、阶段 2（实现）、阶段 3（测试）

**重点关注**：
- 文档间一致性
- 与代码实现的一致性
- 是否有信息丢失

---

### state: research（研究/调研）

**触发条件**：`task_contexts.research`

**激活的规范子集**：
- 阶段 0（理解调研目标）
- 阶段 1（制定调研计划）
- 产出：调研结论（写入对应 doc 或直接在交接块中说明）

**跳过**：阶段 2-4（无代码变更）

**重点关注**：
- 调研结论是否有文档化
- 是否形成了可执行的建议

---

### state: testing（测试相关）

**触发条件**：`task_contexts.testing`

**激活的规范子集**：
- 阶段 0（理解测试目标）
- 阶段 1（制定测试计划）
- 阶段 3（执行测试 + 记录结果）
- 阶段 4（更新 features.md / error_book.md）

**重点关注**：
- 测试覆盖率
- 测试用例是否可重复
- 结果是否记录在案

---

### state: deployment（部署/发布）

**触发条件**：`task_contexts.deployment`

**激活的规范子集**：
- 阶段 0（确认发布内容）
- 阶段 1（发布计划 + 回滚方案）
- 阶段 3（验证发布结果）
- 阶段 4（更新 dev_log.md、project_status.md）

**跳过**：阶段 2（除非发布包含代码变更）

**重点关注**：
- 回滚方案是否就绪
- 监控/告警是否就绪
- 用户通知是否发出

---

### state: cleanup（技术债/清理）

**触发条件**：`task_contexts.cleanup`

**激活的规范子集**：
- 阶段 0（识别清理范围）
- 阶段 1（评估影响 + 制定清理计划）
- 阶段 2（执行清理）
- 阶段 3（验证无破坏）
- 阶段 4（更新 structure.md / features.md）

**重点关注**：
- 是否真的没有功能变更
- 是否引入了新问题

---

### state: performance（性能优化）

**触发条件**：`task_contexts.performance`

**激活的规范子集**：
- 阶段 0（理解性能瓶颈）
- 阶段 1（制定优化计划 + 性能基线）
- 阶段 2（实施优化）
- 阶段 3（重点：性能对比验证）
- 阶段 4（记录优化结果）

**重点关注**：
- 性能基线是否建立
- 优化前后对比是否记录
- 是否有回退风险

---

### state: security（安全相关）

**触发条件**：`task_contexts.security`

**激活的规范子集**：
- 阶段 0（理解安全影响）
- 阶段 1（安全评审 + 修复计划）
- 阶段 2（实施修复）
- 阶段 3（安全验证）
- 阶段 4（更新 CODING_STANDARDS.md 或 security 相关 doc）

**重点关注**：
- 是否引入新的攻击面
- 是否影响现有认证/授权
- 是否需要通知用户

---

### state: data_migration（数据迁移）

**触发条件**：`task_contexts.data_migration`

**激活的规范子集**：
- 阶段 0（理解迁移范围 + 数据结构）
- 阶段 1（制定迁移计划 + 回滚方案 + 数据备份验证）
- 阶段 2（实施迁移脚本）
- 阶段 3（重点：数据完整性验证）
- 阶段 4（更新 dev_log.md）

**重点关注**：
- 数据备份是否可用
- 迁移是否可回滚
- 数据完整性如何验证

---

### state: onboarding（接入新项目）

**触发条件**：`task_contexts.onboarding`

**激活的规范子集**：
- 阶段 0（阅读项目文档：project_status + structure + features）
- 阶段 1（理解项目背景 + 技术栈）
- 阶段 5（产出 onboarding 总结）

**跳过**：阶段 2-4（无变更）

**重点关注**：
- 是否理解了项目全貌
- 是否有遗留问题需要记录

## 状态机执行流程

```
1. 接收任务描述
2. 读取 project_index.yaml
3. 用 trigger_keywords 匹配最合适的 task_context
4. 根据 task_context 查本文件的 state_rules
5. 确定激活哪些规范子集、跳过哪些
6. 按 WCS 流程执行（阶段 0 → 1 → 2 → 3 → 4 → 5，跳过被跳过的）
7. 产出结果和交接
```

## 与 Module 1 的关系

Module 1 定义了"如何判断任务类型"（trigger_keywords → task_context）
Module 2 定义了"判断后如何响应"（task_context → state_rules）

两者共同构成完整的"任务理解与响应"闭环。

## 落地方式

本文件由 WCS skill 在激活时引用，AI 自主决定激活哪个 state_rules。
不需要额外脚本，纯规则驱动。
