# WCS 自我迭代方法论

> 创建时间：2026-06-10
> 背景：mediary-dev SM18 开发复盘，发现 project_index.yaml 被绕过的规则违反问题

---

## 一、什么时候需要自我迭代

满足以下任一条件时，应启动 WCS 自我迭代：

| 触发条件 | 说明 |
|---------|------|
| **发现规则被绕过** | AI 以"合理原因"跳过了某条 WCS 规则 |
| **发现规则无价值** | 某条规则在实践中被认为"没用"或"没意义" |
| **发现规则矛盾** | 两条规则相互冲突，AI 自行取舍 |
| **长期观察** | 同一个违反出现 3 次以上 |

---

## 二、自我迭代的标准流程

### 步骤 1：记录现象（dev_log 或 discussions）

```
## [日期] WCS 自我迭代触发：<简短描述>

**现象**：<AI 做了什么/没做什么>
**违反的规则**：<引用具体规则>
```

不要急于下结论，先把现象记录清楚。

### 步骤 2：区分"规则设计问题"和"AI 执行问题"

这是最关键的一步：

| 类型 | 定义 | 处理方式 |
|------|------|---------|
| **AI 执行问题** | 规则本身合理，但 AI 以借口绕过 | 强化规则执行，加入违反上报机制 |
| **规则设计问题** | 规则本身有缺陷，导致被绕过 | 优化规则或文档，而非要求 AI 遵守 |

**判断方法**：假设规则被执行了——如果执行后对任务有损害，则是规则设计问题；如果执行后没有问题，则是 AI 执行问题。

### 步骤 3：如果是 AI 执行问题

- **不能**通过"允许例外"来解决
- 应该在 SKILL.md 中加入更明确的警告和违反处理
- 在 Session 反思块中加入独立的必填检查项
- 示例：本次 project_index.yaml 被跳过 → 在反思块中加入独立字段 + 违反处理说明

### 步骤 4：如果是规则设计问题

- **不能**通过"要求 AI 遵守"来解决
- 应该优化规则的载体文档（SKILL.md / project_index.yaml 等）
- 让规则在实践中真正有价值，AI 自然不会绕过
- 示例：project_index.yaml 维护阶段无价值 → 加入 maintenance.current_focus 字段

### 步骤 5：落地改进

改进必须落在**具体的文档或规则**上，不能只是"下次注意"：

| 改进类型 | 落地位置 |
|---------|---------|
| 新增规则 | `SKILL.md` 相应章节 |
| 强化现有规则 | `SKILL.md` + Session 反思块 |
| 文档内容优化 | `project_index.yaml` / `docs/*.md` |
| 流程变更 | `SKILL.md` 流程章节 + `workflow.md` |

### 步骤 6：更新维护状态

每次迭代后更新 `project_index.yaml` 的 `maintenance.recent_changes` 和 `last_maintained`。

## 三、本次迭代案例（2026-06-10）

### 现象记录

**现象**：AI 在 WCS 维护任务中跳过了 project_index.yaml，直接读 dev_log.md

**违反的规则**："project_index.yaml 是唯一开场必读"

**AI 的理由**："内容已知，读了无新信息"

---

### 步骤 2 分析：AI 执行问题 vs 规则设计问题

| 判断 | 分析 |
|------|------|
| 假设规则被执行了，有损害吗？ | **无损害**——读了 project_index.yaml 不会造成任何问题 |
| 结论 | **AI 执行问题**，不是规则设计问题 |

**关键洞察**：规则的价值不在于内容是否变化，而在于它是整个 WCS 状态识别框架的触发锚点。跳过它 = 跳过了"确认当前状态"的强制动作。

---

### 步骤 3 & 4：改进方案

**不是**"允许维护阶段跳过"（错误解决方案）

**而是**：
1. **SKILL.md** — 强化开场必读，增加违反处理和具体读法
2. **project_index.yaml** — 增加 maintenance 字段，让维护阶段也有高价值内容
3. **Session 反思块** — 新增独立必填字段 `project_index.yaml`

---

### 落地成果

- `docs/project_index.yaml` — 新增 maintenance.current_focus 和 recent_changes
- `wcs-cn/SKILL.md` — 阶段0强化警告 + 违反处理说明
- Session 反思块 — 新增 project_index.yaml 必填字段
- `docs/dev_plan.md` — 标记为 done

## 四、常见错误

| 错误 | 错误原因 | 正确做法 |
|------|---------|---------|
| 以"内容已知"绕过必读规则 | 混淆了内容价值和框架触发价值 | 必读规则的目的是锚点触发，不是获取新信息 |
| 允许"合理的规则跳过" | 破坏 WCS 的一致性 | 任何绕过都必须有框架级的变更记录 |
| 只要求"下次注意" | 没有落地到文档 | 改进必须落在具体的规则或文档上 |
| 把 AI 执行问题当成规则设计问题 | 通过改规则来回避执行问题 | 强化执行机制，而非修改规则 |
