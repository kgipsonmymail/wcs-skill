# WCS（Wind Code Standard）中文版

## 定位

> WCS 是 AI 开发方法论框架。每个开发任务第一个激活它，通过**状态识别**决定读什么文档，避免上下文过载，在保证开发质量的前提下最快完成任务。WCS 通过在实践中不断总结经验来迭代自身。

## 自迭代规则

**每次迭代开发前必须遵循：**

1. **先 git 备份，再开发新功能** — 任何开发改动前先 `git commit` 备份当前状态
2. **新功能开发完成后，再次 git 备份** — 确保每一步都有可回溯的节点
3. **自迭代时同样遵循 WCS 规则** — WCS 的规则适用于 WCS 自身

**试点项目：**
- **Mediary（日记系统）** — 代码库在 `/www/wwwroot/mediary/`
- **video-daily（视频日历）** — 代码库在 `/www/wwwroot/video-daily/`

WCS 的规则和文档随着这两个项目的实际使用一起迭代，在实战中验证和优化。

## 何时使用

在以下任一情形下触发本技能：

- 在任意仓库中开始新功能
- 修复缺陷或回归问题
- 接续其他 AI 或工程师未完成的工作
- 准备提交或发布交接
- 审计项目文档与实现是否一致
- 接入新项目（onboarding）

## 核心原则：最小化开场，展开式加载

**旧思路**：加载 skill 时把全部 doc 路径塞给 AI，让 AI 自己决定读什么 → 上下文过载

**新思路**：
```
读取 project_index.yaml → 识别状态 → 按状态加载对应 doc → 按需展开下一步
```

### 开场最小化原则

1. **project_index.yaml 是唯一开场必读** — 项目概览，最小信息量。即使是维护任务，也必须先读它确认当前状态，禁止跳过直接改代码或写文档
2. **不要一开始就读 dev_log.md** — 太大了，按需才读
3. **不要一开始就读 architecture / structure 相关文档** — 只有架构相关问题才读
4. **不要一开始就读 errorbook.md** — 只有修 bug 才读
5. **不要一开始就把所有 doc 路径都塞给 AI** — 根据状态按需激活

## 状态识别流程

收到任务后，按以下顺序判断状态：

```
1. 读取项目根目录的 docs/project_index.yaml（开场唯一必读）
2. 根据用户描述中的触发词识别状态
3. 根据状态决定下一步读哪个 doc
4. 读完后再决定是否继续展开下一个 doc
```

### 触发词 → 状态映射

| 用户描述关键词 | 状态 | 说明 |
|---------------|------|------|
| "我要做..."、"计划..."、"打算..."、"新任务" | 计划阶段 | 规划新工作 |
| "写个..."、"实现..."、"开发..."、"做..."、"开始" | 开发阶段 | 编写代码实现 |
| "报错"、"bug"、"坏了"、"崩溃"、"修复..."、"fix" | Bug 阶段 | 修复缺陷 |
| "加个功能..."、"新需求..."、"加一个..."、"增加..." | 需求阶段 | 提出或讨论新需求 |
| "维护..."、"更新依赖..."、"整理..."、"日常" | 维护阶段 | 已有功能维护 |
| "帮我看看代码"、"review"、"评审"、"检查..." | 审查阶段 | 代码或需求评审 |

**注意**：无法匹配时默认使用**开发阶段**流程。

## 各状态文档加载顺序

### 计划阶段

```
project_index.yaml → dev_plan.md → （按需）features.md
```

**何时按需展开**：
- 需要了解现有能力 → 读 features.md
- 需要了解技术限制 → 读 project_status.md

---

### 开发阶段

```
project_index.yaml → CODING_STANDARDS.md → workflow.md → （按需）structure.md
```

**何时按需展开**：
- 需要了解模块结构 → 读 structure.md
- 需要了解架构决策 → 读 architecture.md（若存在）

---

### Bug 阶段

```
project_index.yaml → errorbook.md → （按需）dev_log.md（相关模块章节）
```

**何时按需展开**：
- errorbook 中没有相关记录 → 读 dev_log.md 定位相关模块
- 需要了解历史变更 → 读 dev_log.md 相应章节

---

### 需求阶段

```
project_index.yaml → features.md → dev_plan.md → （按需）project_status.md
```

**何时按需展开**：
- 需要了解现有能力全貌 → 读 features.md
- 需要了解技术约束 → 读 project_status.md

---

### 维护阶段

```
project_index.yaml → 评估现状 → 按评估结果加载对应 doc
```

维护阶段不走固定加载顺序，而是**先评估、再决定读什么**。详见下方独立的维护流程章节。

---

### 审查阶段

```
project_index.yaml → CODING_STANDARDS.md → 待审查代码
```

**审查重点**：
- 是否有规范违反
- 是否有安全隐患
- 是否有性能问题
- 变更范围是否合理

## 开发流程（强制执行）

除非用户明确覆盖，否则执行以下全部规则。

### 阶段 0：初始化与对齐（开场必做）

> ⚠️ **硬性步骤：任何任务都必须先读 project_index.yaml，不允许跳过。**
> 即使是"已知"的项目、即使内容看似无变化，也要读。
> project_index.yaml 是整个 WCS 的状态识别锚点，跳过它等同于跳过整个框架的触发入口。

**强制执行步骤（按顺序）：**

1. **读取 `docs/project_index.yaml`**（开场唯一必读，任何情况下不得跳过）
   - 先读顶部的 `maintenance.current_focus`，快速确认当前维护重点
   - 再读 `task_contexts` 部分，根据用户描述匹配状态
2. 识别当前状态（根据用户描述中的触发词判断）
3. 根据状态加载对应 doc（按各状态的加载顺序）
4. 明确功能/缺陷范围、约束与验收标准

**违反处理**：
- 若未读 project_index.yaml 直接读其他 doc，则视为违反 WCS 规则
- 在 Session 反思块中必须记录：`违反记录：未读 project_index.yaml`

### 阶段 1：开发前置门禁（强制，未完成则停止）

在修改代码前完成全部项：

- 确认已阅读相关规范 doc（CODING_STANDARDS / workflow 等）
- 制定与验收标准绑定的实现计划
- 制定验证计划（单元/手工/集成等，视情况而定）
- 识别将变更的文件与模块
- 对高风险变更准备回滚策略

**若本门禁未完成，停止实现。**

### 阶段 2：按规范实现

- 先实现最小正确解，再优化
- 保持单一职责与命名清晰
- 将环境相关配置外置
- 为变更补充或更新相关测试
- 保持 diff 聚焦；避免无关重构

### 阶段 3：提交前验证

- 对变更范围运行所需测试/检查
- 端到端验证受影响的用户流程
- 确认无阻塞级 lint/类型/构建失败

**验证未通过时不要提交。**

### 阶段 4：验证通过后的文档同步

- `docs/dev_log.md`：改了什么、遇到什么问题、如何验证
- `docs/error_book.md`：根因、失败尝试、最终修复（若涉及缺陷）
- `docs/features.md`：新增/更新的能力条目（若涉及）
- `docs/dev_plan.md`：标记已完成项并细化待办（若涉及）
- `docs/structure.md`：仅当文件/模块结构变化时

### 阶段 5：协作交接

产出简洁交接块，包含：

- 任务范围与结果
- 变更文件
- 已执行的验证
- 已知限制或风险
- 建议的后续动作

## 维护流程（独立流程，与开发流程并列）

> 维护不是"改一下代码然后更新文档"的线性任务。维护是**对项目健康状态的持续治理**，核心是循环：评估 → 规划 → 执行 → 验证 → 再评估。

### 硬性约束（任何维护动作不可违反）

1. **不破坏现有功能** — 维护只做结构调整、文档同步、冗余清理，不改变已有逻辑的行为。重构前确认理解原逻辑，重构后验证功能不变
2. **单文件行数上限** — 不超过 1000 行，理想 500 行以下。超过的文件应考虑拆分模块/提取函数/分离关注点
3. **尊重文件的多元用途** — 不是所有文件都服务于当前线上功能。评估"精简"时必须区分以下类别，**不得将备用/调试/规划类文件误判为冗余代码**：

| 类别 | 说明 | 处理方式 |
|------|------|---------|
| **活跃代码** | 当前生产环境直接使用的代码 | 正常维护 |
| **调试工具** | 开发/排查问题时使用的脚本、诊断工具 | 保留，归入 `scripts/` 或 `tools/`，文档标注用途 |
| **备用方案（Plan B）** | 当前未启用但可能回退或切换的实现 | 保留，文档标注"备用"及启用条件 |
| **规划中（Planned）** | 已写好但尚未接入的代码，为未来功能预留 | 保留，文档标注规划状态 |
| **一次性脚本** | 已完成使命的数据修复/迁移脚本 | 确认不再需要后可归档到 `done/` 或删除 |
| **真正的死代码** | 已被替代、无引用、无备用价值的代码 | 可安全删除，但需 git 历史可恢复 |

**评估流程**：发现"看似冗余"的文件时，先查 git 历史和文档，确认其类别后再决定。不确定时保留并标注，交由用户判断。

### 维护的核心目标（五维模型）

| 维度 | 定义 | 验收标准 |
|------|------|---------|
| **精简** | 去除冗余代码、无用文档、过期依赖、死链接；**但保留备用/调试/规划类文件** | 无真正的死代码；备用/调试文件有文档标注 |
| **有序** | 结构清晰、命名规范、分类合理、层级一致、文件规模可控 | 任何人能在 30 秒内找到任意模块/功能/配置；单文件 ≤500 行 |
| **稳健** | 代码健壮、错误处理完善、依赖安全、测试覆盖 | 无未处理异常，无已知漏洞，核心路径有测试 |
| **文档同步** | 文档与代码时刻一致，无信息孤岛；**已实现功能必须有文档记录** | 改代码必改文档，文档中的代码示例可运行 |
| **高效检索** | 任何信息都能通过索引快速定位 | project_index.yaml 覆盖所有关键入口，无盲区 |

**用户可以指定本次维护的重点维度**（如"主要关注精简和高效检索"），未指定时全部维度平权。

### 维护流程：评估 → 规划 → 执行 → 验证（循环）

```
┌─────────────────────────────────────────────────────┐
│                   维护循环                            │
│                                                     │
│   ① 评估现状 ──→ ② 规划改进 ──→ ③ 执行改进          │
│        ↑                              ↓             │
│        └──────── ④ 验证结果 ←─────────┘             │
│                    ↓                                │
│              ⑤ 同步文档 → 更新索引                    │
│                    ↓                                │
│              是否继续？ ──→ 是 → 回到 ①               │
│                    ↓                                │
│                   否 → 输出维护报告                   │
└─────────────────────────────────────────────────────┘
```

### 操作安全等级（贯穿整个维护流程）

维护过程中所有操作必须标注安全等级，不同等级有不同的执行权限：

| 等级 | 标记 | 说明 | 执行权限 |
|------|------|------|---------|
| **安全** | 🟢 | 只读扫描、创建新文档、追加内容、更新索引 | 可直接执行 |
| **低风险** | 🟡 | 修改文档内容、重命名文件、清理明确的重复项 | 执行前告知用户，无需等待确认 |
| **中风险** | 🟠 | 删除文件、重构代码结构、合并文件 | **必须用户确认后执行** |
| **高风险** | 🔴 | 修改核心业务逻辑、删除活跃代码、改变数据格式 | **必须用户确认 + 备份 + 回滚方案** |

**原则**：评估阶段（①）的所有操作都是🟢安全（只读）。规划阶段（②）输出的任务清单必须标注安全等级。执行阶段（③）遇到未标注的操作时，暂停并询问用户。

### 项目活跃使用中模式

当用户明确表示正在使用项目进行其他工作时，维护流程自动进入**受限模式**：

| 限制 | 说明 |
|------|------|
| **只做🟢安全操作** | 创建文档、更新索引、追加记录——不改任何已有文件内容 |
| **不动代码文件** | .py / .js / .html 文件零修改 |
| **评估照常进行** | 可以完整扫描并输出评估报告，但只记录不执行 |
| **规划照常制定** | 维护计划正常输出，但所有任务标记为"待执行" |
| **用户指定时解除** | 用户说"可以改了"或"开始执行"时，按计划逐项执行 |

**触发条件**：用户说"不要影响正常功能"、"我在用项目"、"先不要改代码"等。
**退出条件**：用户说"可以改了"、"开始执行"、"动手吧"等。

#### ① 评估现状（维护的第一步，不可跳过）

**目的**：先看清项目当前在每个维度上的状态，再决定做什么。

**评估分为两个子阶段**：

##### ①-a 快速扫描（必做）

1. **读取 `project_index.yaml`** — 确认当前维护焦点和文档索引状态
2. **文件规模扫描** — 统计所有源文件行数，标记超过阈值的文件
3. **基线文档检查** — WCS 基线文档是否存在、是否过期
4. **重复文件检测** — 同名/同内容文件是否存在

##### ①-b 深度调研（按需，评估发现问题后展开）

| 调研动作 | 何时需要 | 方法 |
|---------|---------|------|
| **git 历史查证** | 判断文件是活跃/备用/死代码 | `git log --oneline <file>` 查最后修改时间和频率 |
| **路由/API 比对** | 检查文档是否记录了所有端点 | `grep -rn "@.*route" src/` 提取所有端点 vs features.md |
| **import 引用链** | 判断文件是否被使用 | `grep -rn "import.*<module>" src/` 或 `from <module> import` |
| **文档与代码交叉比对** | 检查文档描述是否与代码一致 | 读文档描述 → 读对应代码 → 比对 |
| **依赖关系分析** | 检查是否有过期依赖 | `grep import` 扫描实际使用的库 vs requirements/pyproject |

**调研输出**：对每个"待确认"文件，输出其分类（活跃/调试/备用/规划/一次性/死代码）和判断依据。

##### ①-c 输出评估报告

```
## 维护评估报告

| 维度 | 状态 | 发现的问题 | 优先级 |
|------|------|-----------|--------|
| 精简 | 🟢/🟡/🔴 | <描述> | 高/中/低 |
| 有序 | 🟢/🟡/🔴 | <描述> | 高/中/低 |
| 稳健 | 🟢/🟡/🔴 | <描述> | 高/中/低 |
| 文档同步 | 🟢/🟡/🔴 | <描述> | 高/中/低 |
| 高效检索 | 🟢/🟡/🔴 | <描述> | 高/中/低 |

### 待确认项（需用户决定）
1. <文件/问题> — 分类建议：<备用/调试/一次性/死代码> — 判断依据：<git历史/import关系/文档描述>
2. ...

### 维护焦点建议
<基于评估结果，建议优先处理什么>
```

**优先级判断规则**：
- 🔴 阻塞性问题 → 高优先（如：文档与代码严重不一致导致无法检索）
- 🟡 影响效率 → 中优先（如：命名不规范但不影响功能）
- 🟢 可以接受 → 低优先（如：风格偏好差异）

#### ② 规划改进

基于评估报告，制定维护计划：

1. **选择本次维护范围** — 不求一次解决所有问题，按优先级选 1-3 个维度深入
2. **列出具体任务** — 每个任务有明确的变更目标、验收标准、**安全等级**
3. **识别依赖关系** — 哪些任务必须先做（如：先整理结构，再更新索引）
4. **预估影响范围** — 会改动哪些文件、影响哪些模块

**输出格式**：
```
## 维护计划

### 本次维护范围
- 维度：<选择的维度>
- 目标：<一句话描述>

### 任务清单
1. [ ] <任务描述> → 验收：<验收标准> → 安全等级：🟢/🟡/🟠/🔴
2. [ ] <任务描述> → 验收：<验收标准> → 安全等级：🟢/🟡/🟠/🔴
...

### 依赖关系
- 任务 X 必须在任务 Y 之前完成
- 任务 A 和 B 可以并行

### 预估影响
- 涉及文件：<列表>
- 风险等级：<低/中/高>
```

#### ③ 执行改进

按计划逐项执行，遵循以下原则：

- **先备份** — 执行前 `git commit` 备份当前状态（🟢安全操作除外）
- **安全等级前置** — 执行前确认当前任务的安全等级，🟠🔴 必须用户确认
- **单一任务聚焦** — 一次只做一个任务，完成后再做下一个
- **先验证再继续** — 每个任务完成后立即验证，不要攒到最后
- **遇到新问题先记录** — 执行中发现的新问题记入评估报告，不在当前循环中发散
- **保持可回滚** — 每个🟠🔴任务一个 commit，便于单独回滚

#### ④ 验证结果

每个任务完成后验证：

1. **功能验证** — 改动没有破坏现有功能
2. **维度验证** — 该维度的问题确实被解决了
3. **回归检查** — 没有引入新问题
4. **索引验证** — 如果改了结构，project_index.yaml 是否已更新

#### ⑤ 同步文档与更新索引

验证通过后：

| 文档 | 何时更新 |
|------|---------|
| `dev_log.md` | 记录本次维护改了什么、为什么改、如何验证 |
| `project_index.yaml` | 维护焦点变化时更新 `maintenance.current_focus` |
| `structure.md` | 文件/模块结构变化时 |
| `features.md` | 能力变更时（**新发现的已实现功能必须补充**） |
| `error_book.md` | 涉及缺陷修复时 |
| `project_status.md` | 项目状态变化时 |

**关键规则**：
- 改了代码/结构 → 必须同步更新对应的文档和索引。文档同步不是"事后补充"，是维护流程的一部分。
- 发现已实现但未记录的功能 → 必须补充到 features.md，无论本次维护范围是否包含该模块。

### 维护循环的终止条件

维护是一个可以无限进行的过程，需要明确何时停下来：

| 条件 | 动作 |
|------|------|
| 用户指定的维护目标已达成 | 停止，输出维护报告 |
| 本次评估的所有高优先问题已解决 | 停止，输出维护报告 |
| 用户说"够了"/"先到这里" | 立即停止 |
| 改动开始触及核心功能代码 | 停止，建议转为开发阶段 |

### Errorbook 释放检查（维护阶段特殊机制）

在维护阶段读取 errorbook 后**必须检查**：

**若满足任一条件，在 Session 反思块里输出释放建议**：
- 条目数量 > 20 → 建议释放
- 时间 > 1 个月未释放 → 建议释放
- 用户说"清理 errorbook" → 触发完整释放流程

**释放流程**（详见 `docs/errorbook_release.md`）：
1. 阅读全部条目
2. 分类归纳到对应文档（CODING_STANDARDS / features / workflow 等）
3. 更新 error_book.md（保留仍有效的，标记已归纳的）
4. 在 dev_log.md 记录释放情况

## Skill 管理

### 核心原则：按需激活，不确定时不激活

- 只激活当前任务**明确需要**的 skill
- 不确定时**不激活**，等需要时再说
- 激活后要说明为什么激活

### Skill 分类

| 类型 | 示例 | 何时激活 |
|------|------|---------|
| **规范类** | wcs-cn | 几乎所有开发任务都需要 |
| **调试类** | systematic-debugging | 调试任务、修 bug 时 |
| **项目类** | mediary、video-daily | 特定项目开发时 |
| **辅助类** | writing-plans | 需要制定计划时 |
| **研究类** | 调研类 skill | 研究/调研任务时 |

### 激活判断流程

```
1. 识别当前状态（计划/开发/Bug/需求/维护/审查）
2. 根据状态确定需要的 skill 类型
3. 根据任务描述匹配 skill 的 active_when 关键词
4. 只激活匹配上的 skill，不一股脑全激活
```

### 各状态对应的 Skill 激活建议

| 状态 | 默认激活 | 按需激活 |
|------|---------|---------|
| 计划阶段 | wcs-cn | writing-plans（需要写计划时）|
| 开发阶段 | wcs-cn | 项目类 skill（若涉及特定项目）|
| Bug 阶段 | wcs-cn | systematic-debugging |
| 需求阶段 | wcs-cn | writing-plans（需要写需求文档时）|
| 维护阶段 | wcs-cn | 项目类 skill |
| 审查阶段 | wcs-cn | - |

### 避免上下文干扰

- **不要**一开始就激活所有可能用到的 skill
- **不要**激活与当前任务无关的 skill
- 即使本地有 20 个 skill，当前任务只需要 1 个，那就只激活 1 个

### 自我检查

每次 skill 激活时，问自己：
1. 这个 skill 与当前任务真的相关吗？
2. 当前对话中会不会用到它？
3. 如果不确定，是否可以等需要时再激活？

## 多 AI 协作约定

### 通用约定

- 变更记录写事实，不写聊天记录式流水账
- 明确记录假设，减少上下文丢失
- 文档间优先用链接引用，避免冲突性重复内容
- 规划、实现与状态文档中术语保持一致

### Subagent 监督闭环（强制执行）

当使用 `delegate_task` 工具时，**主 Agent 必须确保**传递的 context 包含以下内容，并在 subagent 返回后执行监督补救：

**传给 Subagent 的 context 必须包含（4个块缺一不可）：**

#### 块1：项目入口文档

```
【项目入口 - 必须首先阅读】
项目路径：<项目绝对路径>
README/CLAUDE.md 内容摘要：<粘贴关键内容>
```

**必须包含的 CLAUDE.md 信息**（粘贴关键段落）：
- 项目简介和技术栈
- 常用命令（后端运行、前端运行、构建）
- Handler 返回格式（如 `{code, message, data}`）
- Git 提交规范
- 关键目录结构

#### 块2：开发规范文档（按需）

```
【开发规范 - 按需阅读】
docs/CODING_STANDARDS.md 要点：<粘贴关键规范>
docs/workflow.md 要点：<粘贴关键流程>
```

#### 块3：任务详细描述

```
【任务目标】
<描述清楚要做什么功能>
<描述清楚验收标准>

【强制完成项 - 退出前必须全部执行】
1. git add -A && git commit -m "<feat: 描述>"
2. 更新 docs/dev_log.md（新功能记录：做了什么、如何验证）
3. 确认无二进制文件被 git 跟踪（ls 确认）
4. 输出 Session 反思块

【禁止项】
- 禁止将 .exe、.log、backend-server 等二进制文件加入 git 跟踪
- 禁止在未 commit 的情况下退出
```

#### 块4：subagent 反思块模板

```
## Subagent 反思
- git commit：是/否，若否则说明原因
- dev_log 更新：是/否，若否则说明原因
- 二进制清理：是/否，若否则说明原因
- 未完成任务：列出因迭代限制未完成的事项
- 改进建议：若有任何规则违反或流程问题
```

**主 Agent 监督流程（subagent 返回后）：**

```
1. 检查 subagent 是否执行了 git commit
   → 若无：主 Agent 执行 git commit + 说明原因
2. 检查 dev_log.md 是否已更新
   → 若无：主 Agent 补充 dev_log 记录
3. 检查是否有二进制文件被误跟踪
   → 若有：清理 + git 恢复
4. 在主 Agent 的 Session 反思块中记录 subagent 执行情况
```

**max_iterations 设置原则：**
- 简单任务（单文件修改）：使用默认 50 次
- 复杂任务（多文件、跨前后端）：**必须设置 100+ 次**
- 若 subagent 报告"接近迭代上限"，主 Agent 立即收回任务

**Subagent 反思块（subagent 必须输出）：**
---

## 本技能的输出约定

触发本技能时产出：

### 状态横幅（每次回复第一行，强制）

```
【阶段: <当前阶段> | 加载: <已加载的 doc 列表>】
```

- **阶段**：当前识别的状态（如 `维护`、`开发`、`Bug`、`需求`、`审查`）

**示例**：
```
【阶段: 维护 | 加载: dev_log.md, dev_plan.md】
```

这样用户在任何时候都能知道：AI 现在处于什么阶段、它的上下文里有哪几个文档在起作用。

---

1. 状态识别结论（当前处于什么阶段）
2. 开发前置门禁清单的执行结果
3. 实现结果与验证摘要
4. 文档同步报告（更新了哪些、为何更新）
5. 可供其他 AI 或工程师接续的最终交接说明

### Session 反思块（强制输出）

每次任务完成后，AI 必须输出一段反思。根据阶段选择对应格式：

**开发/Bug/计划/需求/审查阶段**：

```
## Session 反思

- **状态识别**：[识别了什么状态/是否准确]
- **project_index.yaml**：[是否读了 / 若未读则记录"违反：未读 project_index.yaml"]
- **初始加载**：[开场读了哪些 doc / 是否符合最小化原则]
- **Skill 激活**：[激活了哪些 skill / 是否符合按需激活原则]
- **按需展开**：[后续读了哪些 doc / 是否按状态展开]
- **维护检查**：[errorbook 条目数 / 是否需要释放]
- **Subagent 监督**：[若有使用 delegate_task，记录 subagent 执行情况]
- **本次总结**：[一句话描述本次是否符合 WCS 原则]
```

**维护阶段**（维护有自己的报告格式，反思块聚焦流程合规性）：

```
## Session 反思

- **状态识别**：维护阶段 / 是否准确
- **project_index.yaml**：[是否读了]
- **活跃使用模式**：[是否处于受限模式 / 是否遵守了只读限制]
- **调研深度**：[快速扫描 / 深度调研了哪些维度 / 是否用了 git 历史查证]
- **五维评估**：[是否按五维模型评估了现状 / 评估报告是否输出]
- **待确认项**：[列出了哪些待用户确认的文件/问题 / 用户是否已回复]
- **安全等级执行**：[是否对每个操作标注了安全等级 / 🟠🔴 是否等待了用户确认]
- **维护范围**：[本次选择了哪些维度 / 是否聚焦]
- **执行质量**：[任务完成率 / 是否有中途发散]
- **文档同步**：[改了代码后是否同步更新了文档和索引 / 是否补充了未记录的功能]
- **Errorbook 检查**：[条目数 / 是否需要释放]
- **维护报告**：[是否输出了维护报告]
- **本次总结**：[一句话描述本次维护是否达成目标]
```

**用途**：
- 自我监督 WCS 是否被正确执行
- 用户可查看 AI 是否遵循了最小化原则
- 若 project_index.yaml 未读，则视为 WCS 执行失败
- 若频繁违反，说明规则需要调整
- 作为 WCS 自我迭代的数据来源

**注意**：此反思块对用户可见，作为透明度输出，让用户知道 AI 按什么逻辑工作。
