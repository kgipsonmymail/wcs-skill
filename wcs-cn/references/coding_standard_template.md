# 可移植代码规范模板

在新项目中可将本模板作为 `docs/CODING_STANDARDS.md` 的起点。

## 1. 必须遵守的规则

1. 仅在验证通过后提交
2. 在 `docs/dev_log.md` 中记录实现与验证
3. 在 `docs/error_book.md` 中记录缺陷根因与修复
4. 验证成功后更新受影响的项目文档
5. 文档保持简洁、相互链接、避免重复

## 2. 开发原则

- 先正确，再优化
- 模块/函数保持单一职责
- 优先显式命名与清晰边界
- 环境与配置值外置

## 3. 命名约定

- 文件/模块：按项目约定（`snake_case` 或 `kebab-case`）
- 函数/方法：按语言惯例
- 类/类型：按语言惯例
- 常量：大写下划线（若该语言普遍采用）

如有语言特例，在文档中单独说明。

## 4. 提交约定

格式：

`<type>: <summary>`

类型：

- `feat`：新能力
- `fix`：缺陷修复
- `refactor`：不改变行为的结构调整
- `docs`：文档更新
- `chore`：维护杂项

## 5. 验证策略

提交前：

- 通过相关测试
- 通过变更范围相关的 lint/类型/构建检查
- 验证受影响业务流程

验证未通过时先修复，不要提交。

## 6. 文档同步策略

验证通过后更新：

- `docs/dev_log.md`
- `docs/error_book.md`（如适用）
- `docs/features.md`（若能力变化）
- `docs/project_status.md`（若行为/架构变化）
- `docs/dev_plan.md`（进度与待办）
- 需要时更新 `README.md` 与 `docs/structure.md`

## 7. 协作与交接策略

- 记录保持事实、简洁
- 明确列出假设
- 给出可执行的下一步
- 确保其他 AI/工程师无需隐藏上下文即可接续
