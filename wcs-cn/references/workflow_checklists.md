# WCS 工作流清单

将以下清单作为强制执行的关卡使用。

## 1）接收入口清单

- 确认任务类型：`feature` / `bugfix` / `refactor` / `docs`
- 确认验收标准
- 确认约束（性能、安全、兼容性、截止时间等）
- 确认受影响模块/文件

## 2）开发前置门禁（编码前必须通过）

- 阅读 `docs/CODING_STANDARDS.md`（或等效规范文件）
- 阅读项目工作流与当前状态文档
- 撰写简要实现计划
- 定义验证策略与命令
- 明确交付后需更新的文档

若任一项缺失，暂停编码并补全。

## 3）实现清单

- 变更范围紧扣任务
- 遵循命名与模块职责约定
- 配置外置；避免硬编码敏感信息
- 可行时补充/更新测试
- 除非需求另有说明，否则保持向后兼容

## 4）验证清单

- 运行相关测试命令
- 对变更范围运行相关 lint/类型/构建检查
- 验证核心用户流程或 API 流程
- 在 `docs/dev_log.md` 中记录关键依据

## 5）文档同步清单（验证通过后）

- 更新 `docs/dev_log.md`
- 缺陷或事故类任务更新 `docs/error_book.md`
- 能力变化时更新 `docs/features.md`
- 行为/架构影响时更新 `docs/project_status.md`
- 更新 `docs/dev_plan.md` 的进度与下一步
- 用户可见行为变化时更新 `README.md` 索引或使用说明
- 目录/模块结构变化时更新 `docs/structure.md`

## 6）提交清单

- 确认验证已通过
- 确认必备文档已更新
- 提交信息格式：`<type>: <summary>`
- 保持提交聚焦、便于评审

## 7）交接清单（多 AI 协作）

- 说明目标与完成状态
- 列出变更文件
- 列出验证结果与剩余风险
- 列出待办与建议的下一步
- 记录假设与未决问题
